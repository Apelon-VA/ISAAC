55  	/**
56  	 * Connect to remote service - pull down any existing files in the remote service.
57  	 * <br><br>
58  	 * In general, this operation should not result in any add / commit operations (though implementations may add/commit a 'README' file
59  	 * if necessary to complete the linkage in the case of empty repositories - in fact, are encouraged to ensure that a reasonable README
60  	 * file exists on both the client and the server at the end of this operation)
61  	 * <br><br>
62  	 * This is NOT intended to be called from a running application - rather - it should only be called during the initial configuration 
63  	 * (typically done during the build sequence) as it doesn't provide any feedback as to what local files changed, during its operations.
64  	 * <pre>
65  	 * Implementers should handle the following cases:
66  	 * 
67  	 *  * Remote repository exists, but is empty
68  	 *    o Local folder exists - is not a repository - may or may not contain content.
69  	 *      ==> The expected behavior for this case is that the the local folder will be turned into a repository managed folder, 
70  	 *          and the remote server should be linked to it.  Implementers may commit and push a single file, such as a 'README' file
71  	 *          if necessary - as implementations like GIT and SVN don't allow syncing when there is nothing to sync.  Any other pre-existing
72  	 *          local content should NOT be added, committed or synced as part of this operation.
73  	 *    o Local folder exists, and is already a repository - may or may not contain content.
74  	 *      ==> The expected behavior for this case is that the local repository should be linked to the remote repository.  In the case of GIT, 
75  	 *          it is acceptable to push the local repository directly to the remote server.  This will push any content that was previously committed to the 
76  	 *          repository - but any existing content in the local folder that has not been added or committed should not be pushed to the remote server.  
77  	 *          For other source control systems, this is likely an impossible state - and they may simply discard the local repository information - and link 
78  	 *          to the new location.  Any local content should be preserved, but not committed or added to the remote repository.
79  	 *
80  	 *  * Remote repository exists, and is populated
81  	 *    o Local folder exists - is not a repository - may or may not contain content
82  	 *      ==> The expected behavior for this case is that the server content should be checked out locally - and added to the local folder.  If local files 
83  	 *          already exist that would be overwritten by the checkout (because they also exist on the server) the local files should be PRESERVED - not overwritten.  
84  	 *          The end result of the operation would show these files in a MODIFIED state.
85  	 *    o Local folder exists, and is already a repository - may or may not contain content.
86  	 *      ==> The expected behavior for this case is that the server and local repositories should be linked, and the content from the server should be checked
87  	 *          out and added to the local folder.  The assumption is made that the client and server were a pair at one point - this should be the equivalent of 
88  	 *          updating the client to point to a new URL for the server if the server was relocated, for example.  
89  	 *          If the client repository is found to be incompatible with the server repository - then the client state should be discarded - and a new checkout 
90  	 *          should be done from the server.  Any local files which have naming collisions with server files should be PRESERVED during the checkout - not 
91  	 *          overwritten - leaving them in a MODIFIED state, if they happen to differ from the files that were on the server.
92  	 * </pre>
93  	 * @param remoteAddress - the URL to the remote server
94  	 * @param username - The username to use for remote operations
95  	 * @param password - The password to use for remote operations
96  	 * @throws IOException - Thrown if an error occurs accessing local or remote resources
97  	 * @throws IllegalArgumentException - if the passed parameters are invalid
98  	 * @throws AuthenticationException 
99  	 */
100 	public void linkAndFetchFromRemote(String remoteAddress, String username, String password) throws IllegalArgumentException, IOException, AuthenticationException;

141 	/**
142 	 * Update (all), commit and push the specified files to the remote server.  This can include files that have been modified, added or removed.
143 	 * Assuming that the status call reflects that state.  The implementation will also perform an update from remote as part of this operation.
144 	 * @param commitMessage - the message to attach to this commit
145 	 * @param username - the username to use to push the commit remotely
146 	 * @param password - The password to use to push the commit remotely
147 	 * @param mergeFailOption - (optional - defaults to {@link MergeFailOption#FAIL}) - the action to take if the required update results in a merge conflict.
148 	 * @param files - The list of files to commit.  May be empty, to support cases where a commit was completed, but the push failed due to a merge issue that
149 	 * required resolution.  May be null to request that all tracked files with changes be committed and pushed.
150 	 * @throws IllegalArgumentException - Thrown if an error occurs accessing local or remote resources
151 	 * @throws IOException - if the passed parameters are invalid
152 	 * @throws MergeFailure - If the update cannot be applied cleanly.  The exception will contain the list of files that were changed (cleanly, or not) during the
153 	 * update attempt.
154 	 * @return The set of files that changed during the pull from the server.
155 	 * @throws AuthenticationException

165 	 * @throws IllegalArgumentException - Thrown if an error occurs accessing local or remote resources

175 	/**
176 	 * If {@link #updateCommitAndPush(File, String, String, String, MergeFailOption, String...) or {@link #updateFromRemote(File, String, String, MergeFailOption)}
177 	 * resulted in a MergeFailure exception, this method should be called to specify how to resolve each merge failure.
178 	 * 
179 	 * After calling this, you may call {@link #updateCommitAndPush(File, String, String, String, MergeFailOption, String...)} again.
180 	 * 
181 	 * Note - some implementations (specifically GIT) may throw another {@link MergeFailure} during this operation - this is a secondary merge failure
182 	 * which will also have to be resolved by the user (by calling this method again) before you can commit and push.
183 	 * 
184 	 * @param resolutions - A map where each key is a relative file name of a file that had a mergeFailure, and the corresponding value is
185 	 * the action that should be taken to resolve the issue.  Note that {@link MergeFailOption#FAIL} it not a valid option.
186 	 * @throws IllegalArgumentException - Thrown if an error occurs accessing local or remote resources
187 	 * @throws IOException - if the passed parameters are invalid
188 	 * @throws MergeFailure - If the update cannot be applied cleanly.  The exception will contain the list of files that were changed (cleanly, or not) during the
189 	 * update attempt.
190 	 * @return At a minimum, the set of files modified by the resolution of the merge.  Implementation may, at their choosing , return the complete set of files 
191 	 * that changed during the pull from the server that led to the merge failure, in addition to the files that were changes to resolve the conflicts.
192 	 */
193 	public Set<String> resolveMergeFailures(Map<String, MergeFailOption> resolutions) throws IllegalArgumentException, IOException, MergeFailure;
